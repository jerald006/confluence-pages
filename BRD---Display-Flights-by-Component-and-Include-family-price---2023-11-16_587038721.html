Introduction

When a user makes a Flight search, it is required to display the flight results by components to allow the user to select the more convenient flight as Departure and display the combinable option for the following components.

Purpose

Change the way to select the flights from the consumer to a more descriptive and sensible way for each component.

Example: https://drive.google.com/file/d/15vpTOV-QiWkF_UL_vhWWE-LFd3kuvyRw/view?usp=drive_link

Scope

Context

Technical requirements

  • Create a new wrapper that transforms the Itinerary Cards into a Dictionary of flights

  • Create a node that summarizes the list of possible combinations.

  • Associate the Air pricing info per each flight combination.

  • Create an RPC to call the Wrapper.

  • Change the UI to display flights offered by the component.

  • Once a Flight is selected for a component, display only the combinable flights in the next component based on the selected price.

  • As thomalex does not reply to the fare construction, apply a display logic to split the price of each component (Base Fare and Taxes / Nbr of components)

This is not the standard of the industry, it is only at the UI level to display the price but is not aligned with the right Price Breakdown per component.

Context

The response of Thomalex is returning as offer itineraries, analyzing the response the flight segments are repetitive as many times as the flight segment is combinable in multiple family fares. This represents a lot of duplicated data and limits the visual offer to certain combinations. Flight A only with Flight B per price.

Users Requirement

Pre-condition of the user, Business context for the feature, expected result

  • Applicable to any type of user who searches for Flights.

  • Users can select the departure flight more conveniently on their preferences

  • Filter the list of flights for 2nd components based on the 1st componentÂ´s selection to only the combinable options based on the selected price/family

  • The summary of the individual prices per component is equal to the total price from Thomalex.

Functional Goals

Description of the business impact

  • It has a high visual impact for users who claim a better flight production, giving the users the possibility to select and create the itinerary based on flight preferences instead of selecting the itinerary based on offers.

  • Reducing the amount of data handled by the UI will increase the performance of resolution.

Constraints and Risks

  • The split of the prices is not an industry standard, we need to be sure that we are aligned with the total price of the itinerary received from the source (Thomalex)

  • The split of the prices is an internal math operation to display a price accordingly.

  • Compare the current performance to resolve flight offers vs the new display.
