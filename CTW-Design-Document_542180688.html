Dheeraj’s Questions/Observations

  • SOW has a reference to a communication module which is missing in the design document

  • The sow mentions the following items in the Data Storage section none of which are covered in the design document

    • Data structure of the Database by component

    • Data structure of the Database by role

    • Tables list and list of (some, not all fields) for each table

    • Definitions of API integration methods to be used to connect to the middleware(only existing CTW order /offer API's are mentioned)  

  • Design and wireframes are clearly out of sync

    • Shop workflow has optional fields like 'Flex' , 'Calendar' etc. is missing from the wireframe

    • In the wireframe "Order" screen we have two items "Refresh TST and Split Order" which have no description in the design document

  • The document should also contain the following points:

    • Authorization - Details on how this is to be implemented. How application restricts access to screens, functions and content based on user role? Restrict access to customer information? 

    • Onboarding a new Tenant is a self service operation in XeniApp. What is the basic configuration(e.g. maybe its assigning PCC to tenant) required for the tenant to start booking flights immediately. How would this configuration be setup in the system ?  

    • The document does not describe points of integration with XeniApp application components.

  • User Session Management (e.g. user session timeout, UXL1 user decides kick out a malicious user out of the system with immediate effect or wants to disable a tenant with immediate effect)

  •  Shopping Cart / Exchange Cart / Checkout:

  •  Part of these functions need information from both CTW and XeniApp subsystems how will this be managed. Will CTW build the extensible shopping cart / checkout solution with hooks to add on other inventory type in the cart or the expectation is that CTW expects hooks in the shopping cart / checkout functions build by Xeniapp

  •  How will the checkout transaction across multiple inventories managed?

  •  How are changes to user profile(e.g. tenant pcc modification/ user information change) propagated to items in cart

  • How is existing passenger information retrieved

  • Workflow / screen transition and actions to MW API to CTW Offer Order API mapping and highlight if new API's (not documented at https://api.ctw.direct) are being created. Please provide a summary of the API's function for a new API/ or an API not covered in the existing documentation. E.g.

    • API to email Itinerary to customer

    • API to capture order details in Database

  • API 's implemented by the Admin Backend

  • Physical Architecture

    • How the security module and the MW are made lineally scalable. What are the hardware/external software requirements to accomplish this 

  •  Are there any specific hardware security requirements

  •  What considerations are have been made for fault tolerance/resilience          

    • Proposed Application monitoring 

  • How does Xeniapp team come to know if the security module or the middleware goes down

  • Call out on any dependencies on XeniApp team (application components) , integration api's etc
