XeniApp : Edit a Policy

Created by User d8ec2 on Jun 09, 2021

The customer may want to edit their policy. This can be accommodated through the BHTP API. Information on whether or not a policy can be edited may be discovered by retrieving the policy and checking the canBeEdited property.

Depending on the state of the policy, the date it was purchased and the current date, there are certain things that may or may not be changed.

Suggested Edit flow:

  1. Authenticate to the API

  2. Get the policy with edit information

  3. Check the edit for validity and eligibility

  4. Confirm the edit

2. Get the policy with edit information

This returns the policy along with all the information about what may be changed on the policy.

LocationTry it

GET /v2/policy/edit/{policynumber}

Content-Type: application/json
Authorization: Bearer {your-token.from-the-authentication.response-here}

This endpoint returns all the information about the policy and includes all rules for editing it.

Policy and Coverage Information

  • canAddTravelers: if canBeEdited is true, this indicates if the policy can have travelers added to it.

  • canRemoveTravelers: if canBeEdited is true, this indicates if the policy can have travelers removed from it.

  • canRemoveCoverage: if this is true, the coverage can be removed. If false, the coverage cannot be removed. This is found within each coverage.

Correlation Ids

The response contains correlation Ids for travelers and flights. These ids must be passed in to the Edit Check and Edit Confirm requests as it uniquely identifies these list items between requests.

Rules

The policyRuleSet node contains information about: field-by-field rules and whether they are - required - editable constraints which identify things such as - max and min lengths - available traveler relationships - coverage specific rules * an array of ids identifying which's package's rules were applied

Fields Node

The fields node contains field-by-filed or field group rules as they pertain to what is requested, required, and editable.

the fields node contains three major sub-nodes: policy policyholder * additionalTraveler

Each of these nodes contains the same model structure:

Copied!{
    "propertyname" : {
        "base": ["r", "e"],
        "overrides" : null
    },
}

override node

In the Get for Edit response, override will always be null. This property has use in other places in the API.

base node

The base node contains an array of keys identifying how the field should be configured: if the property name is present, that means the field should be shown "r" means required * "e" means editable

This example shows trip dates are required and are editable

Copied!{
    "tripDates": { "base": ["r", "e"] }
}

This example shows email is optional but cannot be changed

Copied!{
    "email": { "base": [] }
}

Constraints Node

The constraints node contains information about how the data should conform for the edits.

There are 3 major portions of the node: relationship descriptions coverage rules * the rest of the constraints

Relationship Descriptions

The relationships are used to dynamically populate choices for an additional traveler and indicates that traveler's relationship to the policyholder. It lists all relationships available to be covered. Some relationships are not allowed by certain states, in which case, the canSelect value will be false.

The model for each relationship contains: the key to send in the traveler node a display value * whether or not it can be selected/ is allowed by the state (the override property is not used in the Get for Edit endpoint)

Coverage Rules

the coverage rules property is used to convey coverage specific data requirements. The coverage rating Id is the key of the property.

Here is an example of requirements specific for Car Rental Collision Coverage being selected by the user:

Copied!{

            "base" : ["r"],
            "overrides" : null
        },

If the coverage were selected by the user, it would require rental dates to be entered and it would request a car rental company name to be supplied to the policy
