Functional objective

All the reservations created through Flight Product must include the SSR:FOID to avoid any issues at the time of ticketing.

Context

In the Airline industry, every time more Airlines claim the form of identification (FOID) of each passenger in the booking at the time of Ticketing.
We are facing some problems with this rule, since some ticketings failed because of it, and solving the problem requires manual intervention, for reservation through Sabre and Amadeus GDSs.

TravelFusion is not a standard GDS, so they do not claim for FOID. Anyway, the idea is to pass via Thomalex the FOID information for every reservation.

Prerequisites

  • Get the National Identification of each passenger in a booking.

MVP Scope

  • Pass the ID Number to the Thomalex endpoint /cart/book

  • Not pass the ID Type

  • Not Pass the Nationality code

  • Applicable for All reservation created in Xeni

  • Implemented only for GDSs Sabre and Amadeus (TravelFusion not required the FOID inclusion).

The inclusion of extra information will demand to create new commands in Thomalex to send the ID Type as variable and Nationality.
These actions will be face in a further version of the MVP.

Functional flow

Functional Items

Add in the passenger detail page components to get the passenger Identification for each as required.

  • Type of Identification: PP - Passport/ NI - National Identification

  • Identification Id: It is a string that could be only numbers or Numbers and letters.

Split the Passport information and move to a different optional collapsable section right below the Passenger details form.

The identification ID Number must be passed to Thomalex endpoint: POST /api/v1/cart/book.

All the transactions created in Sabre and Amadeus have to hold the FOID at the time of Ticketing.

The passenger additional information - Passport, should be required in case of an International itinerary (Arrival point is in a different country of the departure point)

If the passport is required, the section has to be expanded automatically.

Comments

Update the Passenger Details form in the UI accordingly to the Designs

Add the fields:

Title: Traveler identification.

  • Nationality : Required. Dropdown to pick the country of nationality.

  • Type of Document: Required. Dropdown list. Select, NationalId, Passport.

  • DocumentID: Required. Accept Numbers and Letters. From 6 to 12 chars

2

Split the Passenger Detail Passport Data and move to a new section

Create a new section in the landing, collapsable, that holds the passenger Passport for this MVP

3

Collect the data and pass the passenger identification to Thomalex

  • The documentID only should be passed in the property "Travellers[n].NationalIdentity"

4

Identify type of Trip (Domestic and International)

  • Domestic: The departure and the arrival airport are in the same county in each segment of the itinerary.

  • International: The departure and the arrival airport are in different countries in a segment of the itinerary.

5

Make the “Passport” information required.

If the passport is required for the destination, then the “Passenger Additional Information” has to be expanded and a message indicating “Passenger Passport is required”.

6

Passing Passport Information to Vendor

The same logic as Today.

The new section of “Passenger Additional Information” is for each passenger, and for this MVP will hold only the Passport information, further we need to include there:

  • Frequent Flyer: Up to 3 per passenger and Segment association

  • Special Meals: Each traveler can request special meal for each segment in the itinerary

  • Special Services: Each traveler can request special services per segment like wheel chair, onboard assistance, Baby Crib, etc

  • Travel Documentation: Apart from passport, the user can enter:

    • DOCA: Residence or Destination Address

    • DOCA: Redress Number, Known Traveler ID or VISA information for a certain segment.
