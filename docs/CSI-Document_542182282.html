CSI Watch List Screening REST API

Step1:  CSI support creates a client account, and generates a Client_ID, Client_Secret, Scope and grant_type. CSI Support will then securely e-mail this information to the client’s technical representative

Step2: generate a bearer token using step1 info in postman ( refer image and endpoint details)

 

Step3:  make a WLS request using generated token.

 

WLS API has been created so clients can screen their customers, vendors or other entities against the most current watchlists by using this API. Information such as Name, Identifiers, a unique Identifier, Address information, DOB, and more can be passed to CSI servers to be screened in real time.

 

Note: There are a few new key things to note in the response that help with tracking.

Transaction ID: It is a CSI system generated number that is available in REST responses for the Web Services Watchlist Screening, or in the Watchdog Elite User Interface (UI). The Transaction ID is generated each time a screening occurs. Each screened item will have their own unique transaction ID to use for WLS.

Unique ID: It helps in enabling Good Customer List (GCL) and Blocked Customer List (BCL) when a name and ID are passed through to CSI servers. Unique ID is also required for Customer Data Master (CDM). This is an integration offered by CSI that helps clients continuously scan their database/ entities that have been submitted to CSI.

 

 

 

 

 

 

Bearer Token

Method : Post

Endpoint :

§  Production: https://rcauth.csiweb.com/Connect/Token

§  Test: https://testrcauth.csiweb.com/Connect/Token

Parameters :

§  Client_ID

§  Client_Secret

§  Scope

§  Grant_type

Bearer Token Parameter Definitions

Client_ID: This is the name of the client, used to identify the customer for the authorization process.

Client_Secret:  Much like the GUID /Billingcode in a SOAP action API call, the Authorization call requires the Secret code to help create the authorization Token.

Scope: This specifically indicated the type of request / API. In this case the scope will be set up as “wlsapi”.

grant_type:  Access type, permissions. This will be “client_credentials”
