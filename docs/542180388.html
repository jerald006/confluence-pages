XeniApp : UT(Unit Testing)/Integration Testing

Created by Jaychandra Kushwaha, last modified by Prachi Bhat-Athawale on Nov 30, 2022

UT Rules:

  1. Use userEvent

  2. Take snapshot https://jestjs.io/docs/snapshot-testing#:~:text=Snapshot%20tests%20are%20a%20very,file%20stored%20alongside%20the%20test . 

  3. Mock services

  4. Focus on functionality testing because async, state change already handled by react-testing-library for developer.

  5. We can not mock and call each and every internal function in unit test cases as its part of rendering and calling internally by testing library.

  6. once any async function is called use await or waitFor function to get rendered data with updated dom.

  7. Side Effects handled by library itself.

  8. Write assertion after this try to access DOM element and match with mock data.

  9. Write some different data for different use cases.

  10. Write negative test case to get some more coverage

  11. Every component is single responsibilities so UT will be seprate.

  12. We can do Integration testing if we want additional more coverage(Takes time of developers)

  13. Automation we can implement or BDD on top of UT and Integration by QA Team( will take time and effort)

  14. Need to follow this. https://kentcdodds.com/blog/common-mistakes-with-react-testing-library

  15. Test cases will be applicable if we returning something from function and expecting/asserting from there. Rendering of component according to different data after event, async operations, api request, state change etc… will happens.

  16. Nothing we can achieve if we just calling the function and not expecting/asserting something.

  17. In component code do not put static text like this its mostly bad practice. Even-though we do not want to support multi language, it should be in json file or somewhere in ts file as const or enum

18. Always mocks labels with CAPITAL text so that it will be visible what we are testing.

19. Every function does not have api call or side effect do not use async, if needed then use. For normal use cases just render the component and just write assertion.

20. No Need to write afterEach and beforeEach its already taken care by jest and testing library until we do not mocking any services or library.

21. Write common utility function if needed but be careful not hide/abstract more code in function.

22. Do not try to use datatest-id always try to use role, aria-* etc. If there is no clue then use data-testid

23. For datatest-id we need to follow only one naming convention I recomend hyphen seprated words with lower case.

24. Create seprate file for mockData in mock folder. Use single quote for string.

Note: Developer can take reference from X3UI-452 - Add or Improve Unit Test Cases Done code for 4 components those are basics templates.
