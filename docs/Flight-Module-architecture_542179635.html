XeniApp : Flight Module architecture

Created by Rajesh (Unlicensed), last modified on Jun 14, 2021

  1. Client loads the flights url

    1. the router checks if user is authenticated, only then loads the module

    2. we should have internal router or route changes once the module as long as user is in flights module the url should be /flights

  2. for dropdown look ahead, wait for 500ms or 1 sec after typing complete or paused before triggering the search endpoint, so it will reduce number of calls

  3. once client enters all parameters and clicks on search, currently its using Websockets

    1. there is limitation of 200 simultaneous connections

    2. move static parameters to xeniapp-api and do not hard code on react

    3. so once we load the data we will close connection once we start receiving heartbeats and open a new one when user scrolls down 70% of screen

  4. user selects the flight

    1. send the flight legs to xeniapp-api and which internally makes multiple calls and populate data- rather then client browser making multiple calls

    2. user selecting upgrade options

      1. sends following objects

        1. cabins available for each leg

        2. add unique key for each itinerary offer

        3. matrix of offers to legs to itinerary offer

        4. massage data as required

      2. once user selects cabin for one of the leg

        1. use the matrix above to re-render the cabin selection

        2. display the total using unique key

    3. do not save the cabin selection info to local storage it will break the logic if user goes back and selects different flight

  5. user clicks on add and checkout
