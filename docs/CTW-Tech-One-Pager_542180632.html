Integration proposition Xeniapp/CTW

 

This document will explain in high level the proposed integration between the two systems of Xeniapp and CTW, and the new subsystems and modules that will be created to enhance the existing Air Travel Engine provided by CTW.

 

An integral part of this document is the schema: Integration Diagram Xeniapp.png, see below.

The modules/databases marked with Blue color will be created solely by CTW as part of the SOW agreement. The white components should be created by Xeniapp as part of their existing infrastructure. The green components(UI) will be handled in a separate document.

 

The following SQL and key-value databases will be created:

  • Security DB – a table or a set of SQL tables that will hold all security restrictions. This information will be used to define which user is allowed to make calls to which APIs, the respective volume restrictions and which json preferences can be used. This DB should have a close integration with the Users DB.

  • Markup/CM DB – a table or set of SQL tables that will hold all markups defined by the appropriate users and all content management information (which tenant operates with which PCC etc). This table should have a close integration with the Users DB.

  • PNR stats DB – a table or a set of SQL tables that will record all PNRs for statistical purposes. The purpose of this DB is to hold data used to be shown in the Command Center. This DB should have a close integration with the Users DB.

  • Search Results DB – a key-value database that enables the Middleware to parse all json requests and add/remove values from them. All data in this database will have a short TTL.

 

The following Modules that handle the communication between UI, Xeniapp system and CTW API will be created:

  • Security Module – After the request comes from the UI this module checks if the user has the rights to send such a request or if volume restrictions are met, validates some properties of the json request if needed and sends it to the Middleware Module.

  • Middleware Module – When an Air Travel subsystem request is made by the UI and has passed through the Security Module, the Middleware Module should get information about all appropriate markups/content for the user, attach them to the json request and send that information to the CTW Shopping/Booking engines. When the communication is done upstream, meaning a response is generated from the CTW Shopping/Booking APIs, the Middleware module will record all statistical data needed in the PNR Stats DB and will strip all non-relevant information from the response while using the Search Results Database for short-lived storage and serve it to the UI.

 

The UI project(marked with Green) will be handled in a separate integration which will be more complex and cannot be divided in the same manner as the rest of the Back-end systems described above. The UI project includes both the Customer facing UI and the Command Center.
